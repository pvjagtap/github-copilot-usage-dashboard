Issue #5 — Usage statistics don't match actual numbers from GitHub

Report: github.com/pvjagtap/.../issues/5

User sees 7,347.7 credits in the extension vs 5,278.2 credits reported by GitHub. Subtracting the user's Ollama line (1,662.33 credits) leaves 5,685.4 — close but still ~407 over.

1. Root-cause analysis

1a. Bug — Non-GitHub models (Ollama, BYOK, local) counted as billable (primary)

VS Code's chat & agent surfaces emit OpenTelemetry traces for every model invocation regardless of backend — including Ollama, LM Studio and any "Bring Your Own Key" provider. Our OTLP receiver (src/otelReceiver.ts) ingests them all. In buildDashboardData() any request without copilotUsageNanoAiu (which only GitHub's backend emits) falls back to a rate-table estimate that defaults to gpt-4.1 (200 / 800 credits per 1M tokens) when the model name isn't recognised. The result: Ollama traffic shows up as "AIC credits" and inflates the total.

Confirmed by GitHub's billing docs:

"Code completions and next edit suggestions are not billed in AI credits."
"Copilot features that use AI models consume AI credits" — i.e. only features routed through GitHub's billing backend; BYOK/local models are paid for by the user's own provider key.

1b. Bug — totals span from AIC effective date, not the current billing cycle (secondary)

dashboardData.ts filters aicTurns with t.timestamp.slice(0,10) >= AIC_EFFECTIVE_DATE (= "2026-06-01") and computeSummary() sums every entry it receives. For users whose billing cycle starts mid-month (e.g. signup-anchored cycles), or once we cross into July, the dashboard will keep counting credits beyond the current cycle — so the "monthly budget" reading is wrong.

1c. Gap — Auto-model 10% discount not modelled (minor)

GitHub's doc: "If you are on a paid Copilot plan, you qualify for a 10% discount on model costs while using auto model selection." When copilotUsageNanoAiu is present this is already baked into the billed value. Only the rate-table estimate path misses it. Left as a follow-up because the API-billed value dominates the realistic case.

2. Solution design

2a. "Billable" classifier (the single source of truth)

A credit entry is treated as billable iff either:

  1. It carries an explicit copilotUsageNanoAiu > 0 (= GitHub's backend already billed it), or
  2. Its model name matches an entry in DEFAULT_MODEL_COSTS (a known GitHub-Copilot-supported model), and the model is not in the user-configured excludeModels list.

Everything else (Ollama, LM Studio, BYOK keys, unknown custom models with no nanoAiu) is non-billable — surfaced in the dashboard as an "informational" section but excluded from totalCredits, the budget bar, overage cost, projected total, and the daily-limit guard.

2b. Cycle-window scoping

AICCalculator.computeSummary() now filters every entry to the current billing cycle window (billingCycleStart … billingCycleEnd inclusive). This aligns "dashboard total" with "what GitHub will charge you this month".

2c. New settings

SettingDefaultPurpose
copilotUsage.aic.includeOnlyBilledModels true Master switch. When true (recommended), non-billable models are listed but excluded from totals.
copilotUsage.aic.excludeModels [] Explicit user-supplied substrings to force-exclude (e.g. ["my-internal-llm"]).
copilotUsage.aic.extraBilledModels [] Model substrings to treat as billable even without a rate match (escape hatch for previews).

3. Deviations & trade-offs

  • No name-pattern allow/deny list for BYOK (e.g. no hardcoded /ollama/i). Rationale: the presence of copilotUsageNanoAiu plus the rate-table membership is a sharper signal than provider-name guessing. Avoids false positives like the legitimate raptor-mini fine-tune.
  • Non-billable credits are still estimated and shown (in their own table) so users can see how much their local-model traffic would cost on Copilot — useful for capacity planning. Just not summed into the billed total.
  • 10% auto-discount on rate-estimates not yet applied; the API-exact path already reflects it. Tracked as a follow-up — small effect.

4. Open questions

  • Should the daily-limit guard treat non-billable usage as informational too (recommended) or still pause Copilot when local-model spend would have exceeded the cap? Current behaviour: informational only — it follows aicSummary.totalCredits which is now billable-only.
  • Should auto-detect-plan also flip includeOnlyBilledModels on/off? Currently the new setting is plan-independent.

8. Research — Standard ways to scope "GitHub-billable" usage from logs

Follow-up to issue #5: the v1 fix relies on a substring match against the built-in DEFAULT_MODEL_COSTS rate table as the default arbiter. That heuristic is fragile (new preview models look "unknown" until the table is updated by hand; a BYOK model ID can accidentally substring-match a known family). The question we set out to answer was: what authoritative, documented signals could we use instead?

8a. Signal inventory — ranked by reliability

8b. Signals we looked for but that are not in the debug log

Verified by dumping every attrs.* key on a real llm_request entry from a current debug log. The complete attribute set is: model, debugName, inputTokens, outputTokens, cachedTokens, ttft, responseId, userRequest, inputMessages, maxTokens, requestOptions, requestShape, copilotUsageNanoAiu, systemPromptFile, toolsFile. There is no byok, vendor, endpoint, customModel, or provider hint — so at the debug-log level copilotUsageNanoAiu presence is the only direct signal of GitHub billing. The good news: the Copilot extension only ever writes its own debug log, so every llm_request entry in that file is already scoped to Copilot traffic. Local Ollama / LM Studio called via a different extension never appears there at all.

8c. Why this changes the OTel path too

The OTel receiver listens on 127.0.0.1:14318 — the endpoint the Copilot Chat extension is configured to push to. Spans follow the OpenTelemetry GenAI semantic conventions (gen_ai.request.model, gen_ai.usage.prompt_tokens, …). Crucially, no byok / vendor / billing.multiplier attribute is present on the span — that determination happens later when the API response comes back. So if a user has configured Ollama as a BYOK model inside Copilot Chat itself, the OTel exporter ships a regular GenAI span with the Ollama model id, and the old heuristic had no way to know GitHub didn't bill it. The catalog lookup fixes this: any model id that lives under a non-copilot provider in the CDN manifest (or returns multiplier === 0 from CAPI) is now classified as non-billable up-front, regardless of substring matches against the rate table.

8d. Updated classifier precedence (v1.10.12)

1. config.excludeModels         → force NON-billable
2. hasActualCredits === true    → force billable (nanoAiu from GitHub)
3. config.extraBilledModels     → force billable
4. includeOnlyBilledModels===false → everything is billable (legacy mode)
5. online catalog lookup    → use entry.billable (CAPI > CDN)
6. calculator.isKnownGHCModel() → fallback rate-table heuristic
7. otherwise                    → NON-billable

8e. New setting

copilotUsage.aic.useOnlineModelCatalog (boolean, default true) — gates rule #5. When disabled, the extension operates fully offline using only the built-in rate table (the v1.10.11 behaviour). The catalog is cached for 24h in globalState, hydrated synchronously at activation, then refreshed in the background. Both fetches are best-effort and silent on failure.

  • src/dashboardData.ts — both classifier call sites now pass classifyByCatalog from modelCatalog.ts.

GitHub Copilot CLI (@github/copilot) Live Usage Integration

Goal: add the standalone Copilot CLI as a fourth source alongside VS Code chat sessions, Oh My Pi, and Pi — with the same billing-period filter, the same shared AIC budget, and the same per-source column in the dashboard.

Design decisions (where the spec was ambiguous)

DecisionWhat we choseWhy
Which signal is authoritative for AIC? Hybrid: session.shutdown.data.modelMetrics.{m}.requests.cost when present, else live-walk estimate (prompts × multiplier). The diagnostic in tests/diagnose-copilot-cli.mjs found 4 of 8 sampled sessions had no session.shutdown event (crash, Ctrl-C, still-open). Ledger-only would silently drop ~50% of CLI activity. Live-only is ~5% imprecise on clean sessions because it can't see the CLI's actual rounded cost. Picking the per-model winner reconciles to the GitHub bill on shutdown sessions and recovers ~50% of otherwise-invisible spend on the rest.
Which model do we attribute a prompt to? assistant.message.data.model (most-recent response), falling back to session.start.selectedModel only before the first response of the session lands. Two sampled sessions selected claude-haiku-4-5 in the UI but session.shutdown showed claude-sonnet-4.6 was actually billed (silent fallback / routing). The assistant-side model reflects what GitHub bills.
How is the multiplier resolved? classifyByCatalog(model).multiplier (CAPI live data) wins; built-in FALLBACK_MULTIPLIERS table covers the first scan before the catalog finishes loading. Re-uses the same authoritative source the rest of the extension uses for billable classification — no duplicate rate table to drift out of sync. The fallback table mirrors what's published at docs.github.com/en/copilot/concepts/billing/copilot-requests.
How do CLI credits enter the calculator? creditEntries.push({ actualCredits, billable, model, … tokens: 0 }) — same path OMP/Pi take when they have a known credit value. The CLI bills per prompt × multiplier, not per token. Token fields stay at 0 so they don't inflate the input/output credit subtotals; the calculator sees only the final AIC number.
Slash-command filter Strict regex /^\/[A-Za-z][\w-]*\b/ after trim. Loose startsWith('/') would have matched filesystem paths a user pastes for context (/usr/local/bin/…), undercounting billable prompts.
Reconciliation residual vscodeAicCredits = total − omp − pi − cli The four source columns must sum exactly to the headline AIC total — same invariant the OMP/Pi columns already satisfy.

Deliberate non-additions

  • We do NOT read ~/.copilot/usage.db. Inspection confirmed it indexes VS Code chatSessions (workspaceStorage/…/chatSessions/*.jsonl) — the same source scanner.ts already walks. 0/14 overlap with CLI sessions in the sampled vault. Reading it would cause double-counting.
  • We do NOT read ~/.copilot/session-store.db. It's lazily populated by /chronicle reindex, mostly empty in normal use, and carries no AIC field that isn't already in events.jsonl.
  • We do NOT extend AICCalculator with a prompt×multiplier code path. The existing actualCredits override is already the right primitive — same model OMP/Pi use. Adding a parallel path would create two ways to do the same thing.

Tradeoffs & alternatives considered

  • Ledger-only. Cleaner code, but rejected: the diagnostic proved we'd silently lose half the activity.
  • Live-only. Simpler too, but rejected: when GitHub actually publishes a precise per-session cost, we should honour it instead of substituting our estimate.
  • Treat selectedModel as truth and only show drift to the user. Rejected after seeing the haiku-selected / sonnet-billed sessions in real data — that would over-bill the cheaper model and under-bill the more expensive one. Always attribute to the model that actually answered.
  • Add a fifth source column "BYOK". Out of scope — the existing non-billable models panel already surfaces local Ollama / BYOK runs separately from the billed total.

Open questions

  • Roaming events.jsonl rotation. CLI does not appear to rotate / compact session files. If the file grows multi-MB, the line-by-line re-parse on mtime change will be slow. Current mitigation: 16-way concurrent reads + per-file mtime cache. If we see real-world performance issues, we'd add a tail-only parser that remembers byte offset.
  • Resume-after-shutdown semantics. If a session emits a session.shutdown then a session.resume and more prompts, we accumulate the second shutdown's cost on top of the first (per-model addition). The diagnostic shows this is the intended CLI behaviour, but if GitHub changes how cost is reported on the second shutdown (delta vs cumulative) we'd need to revisit.
  • Silent-reroute over-count. When a session has BOTH a live haiku prompt and a sonnet ledger entry (observed in 2/8 audited sessions — GitHub silently rerouted the model), the per-model hybrid total keeps the haiku live estimate (0.33 AIC) and the sonnet ledger (e.g. 2.0 AIC), so we report 2.33 when GitHub only billed 2.0. Decided to keep the over-count visible rather than silently absorb the haiku prompt — the driftAic field surfaces the delta and lets users investigate. Alternative considered: drop live-only models when hasLedger is true. Rejected because it would hide the routing decision entirely.

Audit pass (post-implementation)

The implementation was audited against the real C:\Users\Jagtprit\.copilot\ vault and a synthetic test fixture. Audit suite at tests/verify-cli-scanner.js — 67 assertions across 4 parts. The audit found 2 real bugs the initial implementation review missed, both now fixed:

  • Bug 1 — slash-command filter dropped filesystem paths. The regex ^\/[A-Za-z][\w-]*\b matched /usr/local/bin/node because \b fires between r and /. User prompts that started with a Unix path were silently excluded from billable counts. Fixed to ^\/[A-Za-z][\w-]*(?:\s|$) — require whitespace or end-of-string after the first token.
  • Bug 2 — session.model_change had no effect on attribution. The handler only updated uiSelectedModel, which is the fallback source. Because lastAssistantModel (set by the prior assistant.message) still won, an explicit user model switch did not take effect until the next assistant response landed — silently mis-attributing the very next prompt. Fixed to update both signals so model_change becomes the immediate ground truth.
  • Latent bug 3 — double-count of paired session formats. 9 of 14 active session IDs in the real vault existed in BOTH <id>.jsonl (legacy flat) AND <id>/events.jsonl (new format). The original enumerateSessionFiles would have visited both. Never reached production because all paired sessions happened to have empty new-format files, but would have triggered the next time the CLI emitted to both. Fixed: mtime-based de-dup with Map<sessionId, Candidate> + zero-byte skip.

Final audit result: 67/67 assertions pass. Real-vault coverage: 23 sessions parsed, 51 billable prompts, 60.99 AIC, 51,103 output tokens, 8 ledger-reconciled / 15 live-only. The scanner's all-time aggregates match an independent re-walk byte-for-byte (Part C.12), and per-session results agree with tests/diagnose-copilot-cli.mjs.
